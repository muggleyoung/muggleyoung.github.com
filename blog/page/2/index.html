回顾–First Project

Dec 4th, 2011

 

 

突然意识到不该再偷懒，当初培养坚持写博的好习惯多么不容易啊，多多少少继续写吧~

工作之后，好多东西顺手就记在了evernote上，乱糟糟一大堆。好处就是，现在随别找点东西改吧改吧就能贴出来了。

下面是在TWU的时候回想自己4个月的工作，回想自己做过的第一个项目的一些记录。
—————————————————————————————–

当我进第一个项目的时候，组里没有QA，而我做为刚入职的新人，对要做的工作近乎于一无所知。项目开发人员耐心解释部署是怎样的过程，教我使用基本的git命令，甚至还要关心我对于一个stroy有没有全面的test case。

但是被开发引导的测试，加上新手tester完全没有测试的sense，这时候我亲爱的buddy出场了！我们不在一个项目组，平时工作遇到问题，首先不方便交流，我咨询组里的同事更快捷，其次，buddy没有我们项目的context，太细节的问题也不好给出答案。于是她安排了一系列的小灶课程，每天午饭后，用半小时的时间跟我介绍QA一般都要做什么，怎么做；敏捷的QA在team中的作用等等，还分享很多资源给我学习。

（注：现在，我们已经有真正的对新入职QA同学的培训了，主要就讲敏捷基础什么的。）

当时的项目并不是很典型的敏捷项目，没有TDD，没有持续集成，没有mingle墙。钱钱在项目总结中说到的好与不好，对我而言只是概念而已，却很难真正理解为什么好，为什么不好，怎样才是好，怎样才是不好。

（昨天还跟同事聊起过，什么理论听起来都特别容易明白，但真正理解到可以清楚明白其中的道理，自己不经历折磨，摸索的过程，是很难真正理解的
现在，虽然我没怎么写过，也不怎么会写TDD，却真正明白了他的重要性，这个聊起后面的项目时候可以说说；还有mingle和物理墙的管理，看似没什么技术含量，但现在也真正明白对于项目，为什么要维护他们，为什么维护总是会被人忘记。）

当时buddy的课程也是被我当作全新的概念在学习，比较少有实践和体会。一个多月的时间，最大的收获是技术的学习，从QA到开发到最后的support阶段，了解了项目中的不同角色都做些什么事情。

这期间，除了快速的上手工作以外，体验support也是难得的经历。紧急修复小bug，处理用户使用中出现的问题，尤其是当系统有问题，客户发飙的时候怎么处理，对自己都是很好的锻炼。第一次被骂时，快哭了的场景，现在想想，也是珍贵的经验了。

最后，感谢又当爹又当妈事无巨细的同事们。耐心的讲解，项目进度不紧张时带我pair，给我机会了解了很多在大项目中，qa接触不到的事情。

话说这搭环境啊

mini项目环境也简单

可出点问题就能晕乎乎的折腾好久，比如说某条命令多了一个空格。。。。

 

一个一个说吧～

tomcat：mac下tomcat安装，貌似有dmg文件，也可以从tomcat主页下载zip包。

安装或解压好之后，vi ~/.profile

set tomcat home:后面的目录就是tomcat所在的位置

export TOMCAT_HOME = /Users/twer/workspace/apache-tomcat-7.0.12

一般启动tomcat都要进入bin目录输入startup.sh，钱钱教我可以设置别名

start_tomcat:

alias start_tomcat = $TOMCAT_HOME/bin/startup.sh

stop_tomcat:

alias stop_tomcat = $TOMCAT_HOME/bin/shutdown.sh

这样一来，可以在任何目录下输入start_tomcat, stop_tomcat启动和停止服务器。省事很多

 

ant：

安装ant，在~/.profile中加入：

export ANT_HOME = /Users/twer/workspace/apache-ant-1.8.2

 

git：

安装git，使用git clone 命令clone一个reporsitory，git pull更新代码

 

IntelliJ IDEA：

安装之后，选择project struction->project->project SDK->new->jdk，找到：system-library-java-javaVirtualMachine-1.6.0.jdk

打开自己的工程就ok

哦，好像还得在profile里设置下project home

 

mysql：

安装没啥说的，设置路径

export PATH = /usr/local/mysql-5.5.11-osx10.5-x86/bin$PATH

导入数据库：mysql -u root targetDatabase < sourceDatabase.sql

导出命令用mysqldump，箭头方向不同而已

 

操作数据库时，我们使用了sequel pro，安装设置好连接参数就行

 

然后！start_tomcat之后，咱的程序就run起来啦！

帮助文档有帮助

May 4th, 2011

上班大约10天时，项目临近上线，开发忙着完成stroy，我这个二把刀测试，除了尽量全面的测试，支持他们的工作，还开始着手写项目帮助文档。

写文档的过程总是痛苦又无聊的。

需要考虑怎样组织逻辑的条理性，包括整个文档结构和你的每一句话；

怎样让用户可以更快捷的找到他们需要的操作；

怎样直观的说明问题，而不是一堆让自己看了都昏昏欲睡的文字；

很多我们看来简单的不能再简单的操作，也要仔细交代；

基于上一条，但有些基础的介绍，几百年都不会有人看，可是万一有不会的，文档里面找不到，就是我们考虑不周到，所以唠唠叨叨说了很多之后，怎样突出操作中关键的逻辑，这也是个问题。

 

幸好咱是刚写过论文的孩只，憋几个文档还是可以忍受的。

近4天时间，还狠变态的花了时间整理字体、排版等面子工程。

当时觉得，这几个文档，算不上好的，也算是above average了。

－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－

项目上线后，一直参与着surprot的工作，突然，就意识到的帮助文档的重要性。

有一类用户，遇到问题会逐步向你说明出现问题时的操作，还体贴的辅以截图。当然我不会一眼看出问题，两眼也看不出，但我经验丰富的pair会。他的经验，加上我测试时对系统的了解，定位问题还是比较迅速的。

另一类用户，由于不了解系统规则，不当操作之后，出现错误，没有预期的结果，他们会莫名其妙，描述问题时，我们也听的狠晕。这时候就多了自己重现问题的工作量。

还经常会出现的问题是，研究了很久，发现系统就是这样的逻辑，只是用户不了解，而我，也不是狠了解。

除了反省自己测试覆盖还不够，对系统需求了解不够多以外，我们是不是需要向用户介绍更多，至少成为这个系统的半个专家。

介绍系统实现的功能，用户操作的基本限制，还要尽量多的说明一些操作上的逻辑－业务逻辑，系统实现上的限制等等。这样可以减少第二类用户出现问题的频率，用户在出错时，可以在帮助文档中找到解决方案，或知道哪里出问题了，交流起来也会更高效。

还有一种方案，就是把具有管理员权限的用户培养成半个专家。其他用户出问题，大多通过求助管理员解决。我们也可以更专注于真正的优化系统，维护完善功能的工作上。

－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－－

用户越是了解系统，我们的维护和后续的开发就会越顺利。

想起刚开始写文档时，愁得不知道该写啥，啥该写

钱钱问我，如果你是用户，你最需要了解什么

我迷惑了一中午。

surport之后，总算有了点体会

其实这个嘛，也可以BA和QA pair完成～嘿嘿～

毕业季-离校

Apr 1st, 2011

总算是彻底毕业了

回来开始办理各种手续，一个人拿着九张离校手续表到处收集盖章

还好咱效率高，大早起来搞完事情，还来得及去门口来个大碗大馍的胡辣汤补补劲儿~

打包宿舍所有东西，没力气打扫，狠心留给阿姨

晃悠去礼堂，坐在角落观摩毕业典礼一小会，本科因为生病错过了自己为自己拨帽穗的时刻，嘿嘿

其实，毕业典礼虽然没劲，有那么一小会，还是情绪激动鼻子发酸

包括退宿舍时，带着阿姨来检查，关门的一瞬意识到，再也不会来这里。。。。

 

昨天又跑去学校领了报到证，事儿就算完了

彻底结束了。。。。

开始打包出差的行李

悲催的发现行李箱有点小，头疼啊

挣钱了再整个！

——————————————————————————–

回到家就病倒鸟，昨天在家睡了半天，带病去学校继续办手续

容易么我，身负这么多人的手续，生病都不能踏实卧床

我2不拉吉的爹竟然还要带我去吃泡馍补补

到泡馍店门口就开始吐。。。。

人啊，还是没逼到那份儿上，就总是很娇气

想之前，只要生病就坚决不下床，睡一整天

现在硬爬下来，虽说吐了一会，该干嘛不还是得去么

而且效率还算高，一小时搞定

感谢同情我的同学们，见娃憔悴，揽下剩下的烂摊子

感谢我爹送我去学校，就是泡馍没吃成很可惜
